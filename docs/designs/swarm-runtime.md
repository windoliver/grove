## Dependency: Issue #132 (Grove v2)

**This design builds ON TOP of #132's architecture, not alongside it.**

#132 delivers the foundational infrastructure this swarm runtime requires:
- **AgentRuntime interface** (spawn, send, close, onIdle) with `AcpxRuntime` default
- **Event-driven routing** via Nexus EventBus — agents get push notifications, no polling
- **System-owned enforcement** on every `grove_contribute` — scores, gates, stop conditions, outcome derivation
- **Automatic stop propagation** — system evaluates stop conditions post-contribute, notifies all agents
- **Workspace = git worktree** with mutation constraints

The swarm runtime adds: experiment orchestration, A/B comparison, benchmark loading, swarm-specific metrics, and the TUI Swarm panel. It does NOT rebuild what #132 provides.

## Vision

### 10x Check
The swarm doesn't just *use* the frontier — it *improves* the frontier. After each benchmark run, the system correlates frontier signal weights with which contributions actually won. Over multiple runs, the frontier self-tunes. Agents that produce higher-quality contributions get spawned more in subsequent rounds — natural selection at the agent configuration level. Every experiment produces a structured, reproducible artifact. Ablation studies become one-liners.

### Platonic Ideal
A researcher types `grove swarm --benchmark swe-bench --agents 50` and within minutes sees a live convergence curve. They pause, inspect any contribution, trace its derives_from chain with diffs at each step, see which agents reviewed it, and understand why the frontier promoted it. The paper writes itself — every experiment exports statistics, scaling curves, and ablation tables. The system feels like a particle physics experiment: configure initial conditions, hit "run", watch emergence.

## Problem Statement

Grove has all the coordination primitives for massively parallel agent work — claims, frontier ranking, derives_from relations, review contributions, bounties, outcomes, and Nexus-backed filesystem-as-IPC. But there's no way to say "spin up 100 agents on this problem and let them self-organize." The protocol is ready for swarms; the orchestration layer doesn't exist yet.

The goal: build the "ant colony" demo where hundreds of agents swarm a problem, review each other's work, build on contributions, and converge — without a central orchestrator.

## Core Claims (revised by CEO review)

1. **Coordination gain, not just scale.** The core claim is that frontier-as-coordinator makes any group of agents better than the same agents working independently. This requires A/B experiments: treatment (with frontier) vs. control (without frontier), same compute budget.

2. **Quality-weighted collaboration depth as metric.** Replace unfalsifiable "emergence" with measurable derives_from chain depth — but only count links where the derived contribution scores HIGHER than its parent. `effective_depth = count of derives_from links where child_score > parent_score`. (Eng review revision — raw depth is gameable.)

3. **Self-organization through protocol.** Agents don't need to know about each other. The contribution DAG and frontier algorithm replace the orchestrator. Research narrative: "What happens when you remove the orchestrator and let agents self-organize through a shared contribution graph?"

### What #132 provides (NOT rebuilt)

| Capability | #132 Component | Swarm Usage |
|------------|---------------|-------------|
| Agent spawning | `AgentRuntime.spawn()` | Spawn N agents per grove with role prompts |
| Push notifications | `AgentRuntime.send()` via EventBus | Treatment agents get notified on new contributions |
| Score validation | Contract enforcement in `contributeOperation` | Benchmark scores validated at write time |
| Gate checks | `metric_improves` gate | Treatment grove auto-flags regressions |
| Stop conditions | Auto stop propagation | Budget (tokens), targetMetric, maxContributions |
| Outcome derivation | Auto from scores + gates | "improved" / "regressed" computed automatically |
| Workspace isolation | Git worktree per agent | Each agent gets isolated workspace |
| Crash recovery | `acpx` persistent sessions | Agents auto-resume after crash |

### Key Design Decisions

- **Agents spawn via `AgentRuntime`** (#132 interface). Default: `AcpxRuntime`. Gives persistent sessions, auto-resume, push notifications. No custom Bun.spawn needed.
- **Roles are prompts, not code.** All agents use the same MCP tools. Role distribution (default 60/25/15 explorer/reviewer/builder) is configurable per experiment for ablation.
- **The frontier IS the coordinator.** No master process decides what agents work on. SwarmOrchestrator manages experiment lifecycle only.
- **A/B via separate groves.** Treatment gets full #132 infrastructure (EventBus routing, frontier, derives_from). Control gets a grove with no event routing, no frontier tool, no derives_from. Zero contamination.
- **Token budget as compute currency.** "Equivalent compute budget" = total input+output tokens. Enforced as a contract stop condition (#132 auto-propagates when budget hit).
- **Two modes: benchmark and open-ended.** `--benchmark swe-bench` loads quantifiable tasks with evaluation requirements in the contract. Without `--benchmark`, agents work on an open-ended task (`--task "improve this module"`) where quality is measured by peer review scores. Both modes support A/B comparison.
- **Event-driven, not polling.** Treatment agents react to contributions via EventBus push (#132 Phase 3). Control agents have no event routing — they must manually query to see others' work (which they can't, since frontier is disabled). This makes the A/B comparison cleaner.
- **Experiment configs are versioned** (schemaVersion: 1) from day 1 for replay compatibility.
- **Frontier weight validation**: weights must sum to 1.0, each >= 0.01. Rejects NaN/degenerate values.

## Implementation Phases (revised for #132 dependency)

### Prerequisites from #132

| #132 Phase | What it delivers | Swarm needs it for |
|------------|-----------------|-------------------|
| Phase 1: Contract enforcement | Score validation, gate checks, auto stop, auto outcomes | Benchmark evaluation, experiment termination |

### Swarm Phase 1 — Prove the claim (after #132 Phases 1-2)

- `grove swarm` CLI command
- SwarmOrchestrator: create experiment, init A/B groves, spawn agents, collect results
- Swarm preset with contract: higher maxActiveClaims, evaluation requirements, stop conditions
- BenchmarkTaskLoader: load SWE-bench tasks into grove
- ExperimentStore: serialize config + results (schemaVersion: 1)
- Basic metrics: solve rate, quality-weighted depth, coordination gain (treatment vs control)
- Open-ended mode: `--task` flag, review consensus as quality signal

### Swarm Phase 2 — Observability

- Provenance tracer (derives_from chain with diffs)
- Agent leaderboard (model/role/prompt performance ranking)
- Cost tracker (per-agent, per-role token accounting)
- TUI Swarm panel (full design spec below)

### Swarm Phase 3 — Automation

- Self-tuning frontier (proper statistical method: holdout, minimum 5 experiments, max 10% weight change, Bayesian update)
- Experiment replay & ablation (`grove swarm replay <id> --agents 100`)

## Success Criteria

- **Measurable**: 50 agents on SWE-bench with frontier outperform 50 agents without frontier, given equivalent token budget
- **Observable**: TUI swarm panel shows real-time convergence
- **Reproducible**: `grove swarm replay` loads identical experiment config
- **Collaborative**: Winning solutions have quality-weighted collaboration depth >= 3
- **Self-improving**: Frontier weights measurably improve across 5+ experiments

## TUI Swarm Panel Design (from /plan-design-review)

### Layout & Information Hierarchy

```
┌─────────────────────────────────────────────────────────────────┐
│ [ SWARM: swe-bench | 42/50 agents | $12.40 | 847k/1M tokens ] │  Header bar
├─────────────────────────────────────────────────────────────────┤
│  CONVERGENCE      ▁▂▃▄▅▅▆▆▇▇▇▇▇▇▇█                          │  PRIMARY
│  best: 0.72  Δ: +0.02/min  depth: 3.4  stall: 0              │  (sparkline)
├─────────────────────────────────────────────────────────────────┤
│  AGENTS  ● explorer-1 working  ● explorer-2 claiming          │  SECONDARY
│          ○ reviewer-1 reviewing ◆ builder-1  deriving          │  (agent grid)
│          (≤20: individual icons | >20: "Explorers: 28/30")    │
├─────────────────────────────────────────────────────────────────┤
│  TOP FRONTIER                                                   │  TERTIARY
│  > blake3:a1b2..  0.72  ⬆depth:3  "Fix parser edge case"     │  (frontier)
│    blake3:c3d4..  0.68  ⬆depth:2  "Refactor token handler"   │
└─────────────────────────────────────────────────────────────────┘
```

- **Sparkline**: Braille characters `▁▂▃▄▅▆▇█`, last 30 data points scaled to panel width
- **Agent icons**: ● explorer (green), ○ reviewer (yellow), ◆ builder (magenta) — shape AND color for a11y
- **Adaptive grid**: Individual agents at ≤20, aggregated by role at >20

### Keyboard Interactions

- `j/k`: Navigate frontier entries
- `Enter`: Open provenance trace (detail mode, full screen, Esc to return)
- `r`: Toggle individual/aggregate agent view
- `s` → `y/n`: Stop swarm with confirmation
- `Tab`: Cycle focus between convergence/agents/frontier sections

### Zoom Behavior

- **Normal**: Full layout (sparkline + agent grid + frontier)
- **Half**: Sparkline + summary metrics + condensed agent counts
- **Full**: Sparkline only with expanded time range

### User Journey

1. `grove swarm` → "Spawning agents..." (agents appear one-by-one via AgentRuntime)
2. First contribution → sparkline starts: ▁ (excitement)
3. First derives_from chain → sparkline rises: ▁▂▃ (wonder — "they're building on each other!")
4. Convergence stabilizes → sparkline plateaus: ▆▇▇█ (satisfaction)
5. Stop condition met → #132 auto-propagates → header: "COMPLETE ✓ | best: 0.72 | depth: 3.4 | cost: $12.40"
6. User traces provenance → full chain with diffs at each step (insight)
